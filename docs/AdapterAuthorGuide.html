Руководство автора адаптера среды

AIStudio — практическая инструкция для разработчика адаптера среды. Версия 0.2 (2026-06-03), contractVersion платформы: 1.0.

Практическая инструкция для разработчика адаптера, у которого установлена та же AIStudio, что и у разработчика симбионта. Студия — единая среда: собрать пакет адаптера → зарегистрировать → настроить проект симбионта → собрать установщик.

Технические форматы описаны в AdapterContract.md. План функций студии — в AdapterPlatform_ImplementationPlan.md.

1. Кто что делает в одной студии

РольТипичные задачи в AIStudio
Разработчик адаптераСборка host (DLL), формирование пакета, «Проверить», при необходимости — публикация ZIP
Разработчик симбионтаНовый проект, гомеостаз, рецепты/триггеры по schema, виртуальные тесты, «Собрать установщик…»
Один человекВсе шаги подряд: частый случай при разработке адаптера «для себя»

Оба направления используют одну установку студии: зарегистрированные пакеты — %ProgramData%\ISIDA\Adapters\, каркас для разработки — %ProgramData%\ISIDA\AdapterPackageTemplates\demo\ (кладёт установщик AIStudio; кнопка «Каркас demo…» открывает этот каталог).

3. Два сценария (оба через студию)

Сценарий A — всё у себя

  1. Собрать host-проект (bin\Debug / Release).
  2. Собрать пакет адаптера (мастер в студии или вручную).
  3. Зарегистрировать → копия в Adapters\{id}\.
  4. Проверить.
  5. Новый проект симбионта с этим AdapterId.
  6. Настройка данных, тесты, Собрать установщик….

Сценарий B — пакет передали коллеге

  1. Автор выкладывает ZIP/папку пакета.
  2. Коллега: Зарегистрировать → Проверить → новый проект → установщик.

Состав пакета в обоих случаях одинаковый.

5. Сборка пакета через студию (план, фаза 1.7)

Мастер «Собрать пакет адаптера…»:

  1. Источник runtime — путь к каталогу сборки host-проекта. Студия копирует *.dll в runtime\.
  2. Метаданные — форма manifest: id, displayName, version, author, contractVersion.
  3. Schema — выбор JSON в проекте адаптера; проверка имён по Contract.
  4. BootData — копия BootData\Environment\ в пакет.
  5. Настройки host — опционально adapter-settings\Settings.xml.
  6. Установщик — .iss → installer\.
  7. Выход — каталог пакета и опционально ZIP.

6. Структура каталога пакета

MyAdapter\
  manifest.json                     ← ОБЯЗАТЕЛЬНО
  schema\                           ← ОБЯЗАТЕЛЬНО
  runtime\                          ← ОБЯЗАТЕЛЬНО (DLL)
  BootData\
    Environment\
      EnvironmentRecipes.yaml       ← ОБЯЗАТЕЛЬНО
      EnvironmentTriggers.yaml      ← ОБЯЗАТЕЛЬНО
  installer\                        ← если сборка установщика из студии
    symbiont.iss
  adapter-settings\                 ← РЕКОМЕНДУЕТСЯ
    Settings.xml
ЭлементОбязательностьНазначение
manifest.jsonДаИдентификатор, версия, пути, contractVersion
schema\ДаПоля редакторов рецептов/триггеров в студии
runtime\ДаDLL и зависимости host-среды
BootData\Environment\ДаОбразцы YAML для нового проекта

* Если установка только вручную — укажите в README пакета.
** Обязательно, если runtime читает пути из XML при первом запуске.

7. Сборка пакета вручную

Используйте, если мастер из раздела 5 ещё не реализован.

7.1. Соберите host-адаптер

Пример: Release или Debug, каталог сборки D:\MyRepo\bin\Debug\.

7.2–7.3. Папка пакета и runtime

Создайте папку пакета (например D:\sld_Adapter\). Скопируйте в runtime\ все DLL без которых host не стартует. Целевая платформа: .NET Framework 4.8. Не кладите исходники .cs, .csproj.

7.4. BootData\Environment\

# EnvironmentRecipes.yaml
recipes: []

# EnvironmentTriggers.yaml
triggers: []

Для осмысленного старта добавьте примеры с валидными adaptive_action_id / influence_action_id.

7.5. Schema

ФайлСодержание
recipe-preconditions.jsonПоля блока preconditions:
recipe-steps.jsonДопустимые type шагов
trigger-filter.jsonФильтр документа/контекста
trigger-detect.jsonПравила detect:
Совет

Каждый key в schema должен поддерживаться вашим runtime при чтении YAML.

7.6. manifest.json

{
  "id": "sld_adapter",
  "displayName": "SolidWorks Adapter (пример)",
  "version": "1.0.0",
  "contractVersion": "1.0",
  "author": "Your Company",
  "bootDataRelativePath": "BootData",
  "installerTemplateRelativePath": "installer/symbiont.iss",
  "schemaVersion": "1.0"
}
ПолеОписание
idКаталог в Adapters\{id}\; только [a-z0-9_-]
contractVersionВерсия контракта платформы; студия 1.x → 1.0
bootDataRelativePathОбычно BootData
installerTemplateRelativePathПуть к .iss от корня пакета

7.7–7.8. adapter-settings и installer

adapter-settings\Settings.xml — шаблон настроек среды исполнения (не проекта симбионта). installer\symbiont.iss — скрипт Inno Setup; студия вызывает ISCC.exe с define-параметрами проекта и адаптера.

8. Регистрация и дальнейшая работа

  1. AIStudio → Адаптеры → Зарегистрировать из папки… или ZIP.
  2. Копия в %ProgramData%\ISIDA\Adapters\{id}\.
  3. Проверить — отчёт OK / предупреждения / ошибки.
  4. Новый проект → выбрать адаптер → BootData из пакета в проект.
  5. Редактирование гомеостаза, виртуальные тесты.
  6. Собрать установщик…

12. Версионирование

  • contractVersion в manifest — согласование с AIStudio (1.0 для первого релиза).
  • version в manifest — ваша версия пакета (1.0.0, …).
  • Новые поля в YAML/schema → новая contractVersion или согласованное расширение.

13. Чеклист перед регистрацией

  • manifest.json с уникальным id
  • contractVersion: 1.0
  • schema\ — все четыре файла (или согласованный минимум)
  • runtime\ — полный набор DLL
  • BootData\Environment\ — оба YAML
  • installer\symbiont.iss (если сборка из студии)
  • adapter-settings\Settings.xml (если host использует)
  • ZIP/папка без мусора (.git, obj)
