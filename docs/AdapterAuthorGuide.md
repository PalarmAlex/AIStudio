# Руководство автора адаптера среды (AIStudio)

Практическая инструкция для **разработчика адаптера**, у которого установлена та же AIStudio, что и у разработчика симбионта. Студия — единая среда: **собрать пакет → зарегистрировать → настроить проект симбионта → (post-MVP) собрать установщик**. Отдельной роли «владелец студии» нет.

Технические форматы — в [`AdapterContract.md`](AdapterContract.md). План функций студии — [`AdapterPlatform_ImplementationPlan.md`](AdapterPlatform_ImplementationPlan.md).

**Версия гайда:** 0.3 (2026-06-06), `contractVersion` платформы: **1.0**.

---

## 1. Кто что делает в одной студии

| Роль | Типичные задачи в AIStudio |
|------|----------------------------|
| **Разработчик адаптера** | Сборка host (DLL), «Создать пакет…» или ручная сборка, «Проверить», публикация ZIP |
| **Разработчик симбионта** | Новый проект, гомеостаз, рецепты/триггеры (поля из `schema\`), виртуальные тесты |
| **Один человек** | Все шаги подряд: частый случай при разработке адаптера «для себя» |

Оба направления используют **одну** установку студии:

| Каталог | Назначение |
|---------|------------|
| `%ProgramData%\ISIDA\Adapters\` | Зарегистрированные пакеты `{id}\` |
| `%ProgramData%\ISIDA\AdapterPackageTemplates\demo\` | Каркас demo (кладёт установщик AIStudio) |

Меню: **Проект → Зарегистрированные пакеты…**

## 2. Каркас пакета (`demo`)

Готовый скелет с manifest, BootData, **заполненными образцами schema** и каталогом `runtime\`:

| Где | Путь |
|-----|------|
| Исходник в репозитории | `docs/adapter-package-example/demo/` |
| На машине пользователя | `%ProgramData%\ISIDA\AdapterPackageTemplates\demo\` |

**Не регистрируйте** каталог demo напрямую. Используйте один из путей:

1. **Рекомендуется:** **Проект → Зарегистрированные пакеты… → Создать пакет…** — студия копирует demo, подставляет runtime из `bin\Debug`, проверяет пакет и регистрирует в Adapters.
2. **Вручную:** скопируйте demo, заполните `runtime\` и при необходимости отредактируйте `schema\`, затем **Зарегистрировать из папки…**.

Эталон с образцами Velum — `docs/adapter-package-example/velum/`.

## 3. Два сценария (оба через студию)

### Сценарий A — всё у себя

1. Собрать host-проект (`bin\Debug` / Release).
2. **Создать пакет…** (§ 5.1) или собрать пакет **вручную** (§ 7).
3. **Проверить** зарегистрированную копию в Adapters.
4. **Проект → Создать проект симбионта** — опционально выбрать тип среды → BootData копируется в проект, `AdapterId` записывается в `AgentProperties.dat`.
5. Гомеостаз, рецепты/триггеры, виртуальные тесты.
6. При необходимости сменить тип среды: **Свойства симбионта** → ComboBox «Тип среды» (§ 9).
7. (Post-MVP) **Собрать установщик…**

### Сценарий B — пакет передали коллеге

1. Автор выкладывает ZIP/папку готового пакета.
2. Коллега: **Зарегистрировать из ZIP…** → **Проверить** → новый проект → работа с данными.

Состав пакета в обоих случаях **одинаковый** (§ 6).

## 4. Раздел «Зарегистрированные пакеты среды»

| Действие | Статус | Назначение |
|----------|--------|------------|
| **Создать пакет…** | **Реализовано (MVP)** | Manifest → каталог `bin\Debug` с DLL → валидация → регистрация в `Adapters\{id}\` |
| **Зарегистрировать из папки…** | Реализовано | Готовый пакет; валидация **до** копирования |
| **Зарегистрировать из ZIP…** | Реализовано | То же из архива |
| **Проверить** | Реализовано | Manifest, BootData YAML, schema JSON, runtime DLL |
| **Обновить список** | Реализовано | Перечитать каталог Adapters |
| **Открыть каталог Adapters** | Реализовано | Проводник Windows |
| Руководство автора | Реализовано | Этот документ из студии |
| **Собрать пакет адаптера…** (полный мастер) | *План: фаза 1.7* | BootData/ISS из репозитория host, ZIP, профиль в хабе |
| **Собрать установщик…** | *План: фаза 5* | ISCC + dist |

## 5. Сборка пакета в студии

### 5.1. «Создать пакет…» (реализовано)

**Проект → Зарегистрированные пакеты… → Создать пакет…**

1. Форма **manifest**: `id`, `displayName`, `version`, `author`, `contractVersion` (должен быть `1.0`).
2. Диалог **каталога сборки host** (например `D:\MyHost\bin\Debug\`) — обязателен. Студия копирует все `*.dll` в `runtime\` будущего пакета.
3. Студия собирает пакет из каркаса demo + runtime, запускает **«Проверить»** и только при успехе копирует в `%ProgramData%\ISIDA\Adapters\{id}\`.

Если валидация не прошла, пакет **не регистрируется** — исправьте ошибки по отчёту и повторите.

### 5.2. Полный мастер «Собрать пакет адаптера…» (план, фаза 1.7)

Расширение MVP-мастера:

1. Выбор BootData из репозитория host (не только demo).
2. Копирование `adapter-settings\`, `installer\*.iss`.
3. Выход в произвольную папку или ZIP без немедленной регистрации.
4. Сохранение профиля путей в хабе студии.

После появления полного мастера ручная сборка (§ 7) остаётся запасным вариантом.

## 6. Структура каталога пакета (чеклист)

```
MyAdapter\
  manifest.json                     ← ОБЯЗАТЕЛЬНО
  schema\                           ← ОБЯЗАТЕЛЬНО (JSON для редакторов «Среда»)
  runtime\                          ← ОБЯЗАТЕЛЬНО (DLL и зависимости)
  BootData\
    Environment\
      EnvironmentRecipes.yaml       ← ОБЯЗАТЕЛЬНО (допускается recipes: [])
      EnvironmentTriggers.yaml      ← ОБЯЗАТЕЛЬНО (допускается triggers: [])
  installer\                        ← если сборка установщика из студии (post-MVP)
    symbiont.iss
  adapter-settings\                 ← РЕКОМЕНДУЕТСЯ для host
    Settings.xml
```

| Элемент | Обязательность | Назначение |
|---------|----------------|------------|
| `manifest.json` | Да | `id`, версия, `contractVersion`, пути внутри пакета |
| `schema\` | Да | Какие поля показывают редакторы рецептов/триггеров (`AdapterSchemaLoader`) |
| `runtime\` | Да | DLL host-среды на машине пользователя |
| `BootData\Environment\` | Да | Образцы YAML; seed при создании/смене типа среды в проекте |
| `installer\` | Да* | Шаблон Inno Setup |
| `adapter-settings\` | Нет** | Шаблон настроек **host-среды** (не проекта симбионта) |

\* Если установка только вручную — укажите в README пакета.  
\*\* Обязательно, если runtime читает пути из XML при первом запуске (как Velum).

## 7. Сборка пакета вручную

Используйте, если нужен полный контроль над содержимым или готовите ZIP **без** немедленной регистрации в Adapters.

### 7.1. Соберите host-адаптер

Пример (SolidWorks / Velum): каталог сборки `D:\MyRepo\bin\Debug\`.

### 7.2. Создайте папку пакета

Например `D:\sld_Adapter\`.

### 7.3. Скопируйте DLL в `runtime\`

- Все DLL, без которых host не стартует: ваша сборка, `isida.dll`, interop среды и т.д.
- **Не** кладите в `runtime\`: исходники `.cs`, `.csproj`, файлы студии.
- Целевая платформа: **.NET Framework 4.8**.
- `.pdb` — по желанию.

### 7.4. Подготовьте `BootData\Environment\`

**`EnvironmentRecipes.yaml`**

```yaml
# Рецепты среды (привязка к adaptive_action_id в Data агента).
recipes: []
```

**`EnvironmentTriggers.yaml`**

```yaml
# Триггеры среды: событие → influence_action_id.
triggers: []
```

Для осмысленного старта добавьте 1–2 примера с валидными `adaptive_action_id` / `influence_action_id` из `.dat` проекта.

Формат — [`AdapterContract.md`](AdapterContract.md) § YAML среды.

### 7.5. Заполните `schema\`

Студия читает JSON из `Adapters\{id}\schema\` и строит UI редакторов «Среда» (типы шагов, preconditions, detect kinds). Имена файлов для `contractVersion` 1.0:

| Файл | Содержание |
|------|------------|
| `recipe-preconditions.json` | Поля блока `preconditions:` рецепта |
| `recipe-steps.json` | Допустимые `type` шагов (`stepTypes`) |
| `trigger-filter.json` | Фильтр документа/контекста триггера |
| `trigger-detect.json` | Допустимые `detect.kind` (`detectKinds`) |

Пример preconditions:

```json
{
  "schemaVersion": "1.0",
  "fields": [
    {
      "key": "document_kinds",
      "label": "Типы документа",
      "type": "stringList",
      "enumValues": ["part", "assembly", "drawing"],
      "required": false,
      "default": []
    },
    {
      "key": "not_sketch_edit",
      "label": "Не в режиме эскиза",
      "type": "bool",
      "required": false,
      "default": false
    }
  ]
}
```

Пример step types:

```json
{
  "schemaVersion": "1.0",
  "stepTypes": [
    { "type": "set_property", "label": "Установить свойство" },
    { "type": "run_sw_command", "label": "Команда SolidWorks" }
  ]
}
```

**Правило:** каждый `key` / `type` в schema должен поддерживаться **вашим runtime** при чтении YAML. Образцы для demo — в `docs/adapter-package-example/demo/schema/`.

Если schema отсутствует или пуста, студия использует Velum-like defaults (fallback).

### 7.6. Создайте `manifest.json`

```json
{
  "id": "sld_adapter",
  "displayName": "SolidWorks Adapter (пример)",
  "version": "1.0.0",
  "contractVersion": "1.0",
  "author": "Your Company",
  "bootDataRelativePath": "BootData",
  "installerTemplateRelativePath": "installer/symbiont.iss",
  "schemaVersion": "1.0"
}
```

| Поле | Описание |
|------|----------|
| `id` | Каталог в `Adapters\{id}\`; только `[a-z0-9_-]` |
| `contractVersion` | Версия контракта платформы; студия 1.x → `1.0` |
| `bootDataRelativePath` | Обычно `BootData` |
| `installerTemplateRelativePath` | Путь к `.iss` от корня пакета (post-MVP) |

### 7.7. `adapter-settings\Settings.xml` (если нужен host)

Шаблон настроек **среды исполнения** на целевой машине. **Не** путать с `Settings\Settings.xml` проекта симбионта.

### 7.8. `installer\symbiont.iss` (post-MVP)

Скрипт Inno Setup. Студия вызывает `ISCC.exe` с define-параметрами:

- `/DProjectRoot=...` — корень **проекта симбионта**;
- `/DDeployDataRoot=`, `/DDeployBootData=`, … — из `Settings.xml` **проекта** (пути);
- `/DAdapterId=...` — из **`Data\Gomeostas\AgentProperties.dat`** (`AdapterId|…`), **не** из Settings.xml;
- `/DRuntimeSource=...` — из зарегистрированного пакета в Adapters.

## 8. Регистрация и работа с проектом симбионта

### 8.1. Регистрация пакета

1. **Проект → Зарегистрированные пакеты…**
2. **Создать пакет…** / **Зарегистрировать из папки…** / **из ZIP…**
3. Копия в `%ProgramData%\ISIDA\Adapters\{id}\`
4. **Проверить** — отчёт OK / предупреждения / ошибки (§ 11)

Повторная регистрация того же `id` — замена каталога (с подтверждением).

### 8.2. Проект симбионта и тип среды

1. **Проект → Создать проект симбионта** — опционально выбрать тип среды из списка **зарегистрированных** пакетов.
2. BootData из `Adapters\{id}\BootData\` копируется в `{project}\BootData\` (существующие файлы не перезаписываются).
3. **`AdapterId`** записывается в `{project}\Data\Gomeostas\AgentProperties.dat`:

   ```
   AdapterId|velum
   ```

4. Редактирование гомеостаза и меню **«Среда»** (рецепты/триггеры) доступно, если:
   - в свойствах симбионта выбран тип среды;
   - пакет `{id}` **существует** в Adapters;
   - симбионт на **стадии 0**, пульсация выключена.

5. **Смена типа среды:** меню симбионта → **Свойства симбионта** → ComboBox «Тип среды» или «(Без адаптера)». При смене студия может предложить **дополнить BootData** образцами из нового пакета.

6. **Legacy:** если в старом проекте `<AdapterId>` был в `Settings.xml`, при открытии он **переносится** в `AgentProperties.dat` и удаляется из Settings.

7. **Валидация:** если в dat указан `id`, которого нет в Adapters, студия **сбрасывает** привязку и показывает предупреждение.

## 9. Три уровня настроек (частая ошибка)

| Файл | Где | Кто редактирует | Зачем |
|------|-----|-----------------|-------|
| `adapter-settings\Settings.xml` | В **пакете** адаптера | Автор адаптера | Дефолты host-среды после установки у конечного пользователя |
| `Settings\Settings.xml` | В **проекте** симбионта | Студия | Пути Data, BootData, Logs, шаблон ISS (post-MVP); **без AdapterId** |
| `AgentProperties.dat` | `{project}\Data\Gomeostas\` | UI «Свойства симбионта» | **AdapterId**, имя, стадия, промпт; для ISS `/DAdapterId=...` |

Студия **не** подменяет `adapter-settings` при редактировании симбионта.

**Терминология в UI:** «зарегистрировать пакет», «тип среды» — не «установить адаптер» и не «подключить к SolidWorks».

## 10. Редакторы «Среда» и schema

Редакторы **Рецепты среды** и **Триггеры среды**:

- читают и пишут YAML в `{project}\BootData\Environment\` (runtime contract);
- загружают **schema** из `Adapters\{AdapterId}\schema\` через `AdapterSchemaLoader`;
- показывают только поля и типы шагов, описанные в schema (если schema пуста — Velum-like defaults).

При смене `AdapterId` в свойствах симбионта перезагрузите редактор «Среда», чтобы подтянулась новая schema.

## 11. Что делает «Проверить»

| Проверка | Ошибка, если |
|----------|----------------|
| `manifest.json` читается, `contractVersion` поддерживается | Нет файла, неверный JSON, неизвестная версия |
| `id` валиден | Пустой id, недопустимые символы |
| Файлы `schema\` парсятся как JSON | Отсутствует файл, битый JSON |
| `BootData\Environment\*.yaml` читаются кодеком | Синтаксис YAML, нет секций `recipes` / `triggers` |
| `runtime\` не пустой | Нет DLL |
| `installer\*.iss` существует, если указан в manifest | Путь битый |

При **«Зарегистрировать из папки/ZIP»** и **«Создать пакет…»** проверка выполняется **до** копирования в Adapters.

## 12. Частые ошибки

| Симптом | Вероятная причина | Что сделать |
|---------|-------------------|-------------|
| «Неподдерживаемый contractVersion» | В manifest версия новее студии | Поставить `1.0` или обновить студию |
| «Нет секции recipes/triggers» | Неверные имена YAML | `EnvironmentRecipes.yaml`, `EnvironmentTriggers.yaml` |
| «Пустой runtime» | DLL не скопированы | Указать `bin\Debug` в «Создать пакет…» или заполнить `runtime\` вручную |
| Пакет не зарегистрировался | Ошибка валидации до копирования | Исправить по отчёту «Проверить» |
| Меню «Среда» недоступно | Нет типа среды в свойствах симбионта | Выбрать пакет в «Свойства симбионта» |
| «Среда» была, пропала | Пакет удалён из Adapters | Зарегистрировать снова или сбросить тип среды |
| Рецепты OK в студии, в среде нет | Runtime не читает те же ключи | Сверить schema, YAML и runtime |
| Путают Settings проекта и host | Разные файлы | См. § 9 |

## 13. Версионирование

- **`contractVersion` в manifest** — согласование с AIStudio (`1.0` для текущего релиза).
- **`version` в manifest** — версия вашего пакета (`1.0.0`, …).
- Новые поля в YAML или schema → новая **`contractVersion`** или согласованное расширение (см. Contract).

## 14. Чеклист перед регистрацией / публикацией

- [ ] `manifest.json` с уникальным `id`
- [ ] `contractVersion`: `1.0`
- [ ] `schema\` — четыре JSON-файла с осмысленным содержимым
- [ ] `runtime\` — полный набор DLL (проверено «Проверить» или «Создать пакет…»)
- [ ] `BootData\Environment\` — оба YAML
- [ ] `installer\symbiont.iss` (если post-MVP сборка из студии)
- [ ] `adapter-settings\Settings.xml` (если host использует)
- [ ] Пример YAML проходит runtime на тестовой машине
- [ ] ZIP/папка без мусора (`.git`, `obj`)
- [ ] У коллеги та же `contractVersion`, что у его версии студии
