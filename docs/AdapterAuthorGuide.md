# Руководство автора адаптера среды (AIStudio)

Практическая инструкция для **разработчика адаптера**, у которого установлена та же AIStudio, что и у разработчика симбионта. Студия — единая среда: **собрать пакет адаптера → зарегистрировать → настроить проект симбионта → собрать установщик**. Отдельной роли «владелец студии» нет.

Технические форматы — в [`AdapterContract.md`](AdapterContract.md). План функций студии — [`AdapterPlatform_ImplementationPlan.md`](AdapterPlatform_ImplementationPlan.md).

**Версия гайда:** черновик 0.2 (2026-06-03), `contractVersion` платформы: **1.0** (целевое).

---

## 1. Кто что делает в одной студии

| Роль | Типичные задачи в AIStudio |
|------|----------------------------|
| **Разработчик адаптера** | Сборка host (DLL), формирование пакета, «Проверить», при необходимости — публикация ZIP |
| **Разработчик симбионта** | Новый проект, гомеостаз, рецепты/триггеры по schema, виртуальные тесты, «Собрать установщик…» |
| **Один человек** | Все шаги подряд: частый случай при разработке адаптера «для себя» |

Оба направления используют **одну** установку студии и один каталог зарегистрированных адаптеров `%ProgramData%\ISIDA\Adapters\`.

---

## 2. Два сценария (оба через студию)

### Сценарий A — всё у себя

1. Собрать host-проект (`bin\Debug` / Release).
2. **Собрать пакет адаптера** (мастер в студии — § 4, или вручную — § 6).
3. **Установить адаптер…** → копия в `Adapters\{id}\` (можно сразу свой же пакет).
4. **Проверить**.
5. **Новый проект симбионта** с этим `AdapterId`.
6. Настройка данных, тесты, **Собрать установщик…**.

### Сценарий B — пакет передали коллеге

1. Автор выкладывает ZIP/папку пакета (после § 4 или § 6).
2. Коллега в своей студии: **Установить адаптер…** → **Проверить** → новый проект → установщик.

Состав пакета в обоих случаях **одинаковый** (§ 5).

## 3. Раздел «Адаптеры» в студии (целевой функционал)

| Действие | Статус | Назначение |
|----------|--------|------------|
| **Собрать пакет адаптера…** | *План: фаза 1.7* | Мастер: указать `bin\Debug` (или папку сборки) → заполнить manifest/schema → подтянуть BootData, Settings, ISS из проекта адаптера → выход: готовая папка/ZIP |
| **Установить адаптер…** | *План: фаза 1.4* | Копия пакета в `Adapters\{id}\` |
| **Проверить** | *План: фаза 1.5* | Валидация зарегистрированной копии |
| Список адаптеров | *План: фаза 1.4* | Установленные `id`, версии |
| Руководство автора | *План* | Открыть этот документ из студии |

До реализации мастера (§ 4) пакет собирают **вручную** по § 6.

## 4. Сборка пакета через студию (план, фаза 1.7)

Мастер **«Собрать пакет адаптера…»** (имя может уточняться при реализации):

1. **Источник runtime** — путь к каталогу сборки host-проекта (например `D:\MyHost\bin\Debug\`). Студия копирует `*.dll` (и при необходимости выбранные сопутствующие файлы) в `runtime\` будущего пакета. Фильтр: без `.pdb` по умолчанию, опционально с pdb.
2. **Метаданные** — форма manifest: `id`, `displayName`, `version`, `author`, `contractVersion`.
3. **Schema** — выбор существующих JSON в проекте адаптера или копирование из шаблона; проверка имён файлов по Contract.
4. **BootData** — путь к `BootData\Environment\` в репозитории адаптера (или к уже подготовленным YAML) → копия в пакет.
5. **Настройки host** — опционально: `Settings.xml` из проекта адаптера → `adapter-settings\Settings.xml`.
6. **Установщик** — путь к `.iss` в проекте адаптера → `installer\`.
7. **Выход** — каталог пакета (например `D:\Out\sld_Adapter\`) и опционально ZIP; сразу кнопка «Зарегистрировать этот пакет» (вызов «Установить…» без повторного выбора папки).

Профиль мастера можно **сохранить** (пути к проекту host, шаблон schema) в хабе студии — отдельно от профиля проекта симбионта.

После сборки — § 7 (регистрация и проект симбионта).

## 5. Структура каталога пакета (чеклист)

```
MyAdapter\                          ← имя до регистрации — любое
  manifest.json                     ← ОБЯЗАТЕЛЬНО
  schema\                           ← ОБЯЗАТЕЛЬНО (JSON для редакторов студии)
  runtime\                          ← ОБЯЗАТЕЛЬНО (DLL и зависимости)
  BootData\                         ← ОБЯЗАТЕЛЬНО
    Environment\                    ← ОБЯЗАТЕЛЬНО
      EnvironmentRecipes.yaml       ← ОБЯЗАТЕЛЬНО (допускается пустой каталог)
      EnvironmentTriggers.yaml      ← ОБЯЗАТЕЛЬНО (допускается пустой каталог)
  installer\                        ← ОБЯЗАТЕЛЬНО, если симбионт ставится через Inno Setup из студии
    symbiont.iss                    ← или другое имя, указанное в manifest
  adapter-settings\                 ← РЕКОМЕНДУЕТСЯ, если у среды есть свой Settings.xml
    Settings.xml
```

| Элемент | Обязательность | Назначение |
|---------|----------------|------------|
| `manifest.json` | Да | Идентификатор, версия, пути внутри пакета, `contractVersion` |
| `schema\` | Да | Какие поля показывает студия в редакторах рецептов/триггеров |
| `runtime\` | Да | Всё, что нужно host-среде на машине пользователя (DLL) |
| `BootData\Environment\` | Да | Образцы YAML; при новом проекте копируются в проект симбионта |
| `installer\` | Да* | Шаблон Inno Setup для «Собрать установщик…» |
| `adapter-settings\` | Нет** | Шаблон настроек **host-среды** (не путать с настройками проекта симбионта) |

\* Если установка только вручную — укажите это в README пакета; для сборки установщика из студии нужен `.iss` в пакете.  
\*\* Обязательно, если runtime читает пути из XML при первом запуске (как Velum).

## 6. Сборка пакета вручную (до появления мастера в студии)

Используйте, если мастер **§ 4** ещё не реализован. После появления мастера ручная сборка остаётся запасным вариантом.

### 6.1. Соберите host-адаптер

Пример (SolidWorks / Velum): Release или Debug, получите каталог сборки, например:

`D:\MyRepo\bin\Debug\`

### 6.2. Создайте папку пакета

Например `D:\sld_Adapter\`.

### 6.3. Скопируйте DLL в `runtime\`

- Все DLL, без которых host не стартует: ваша сборка, `isida.dll` и зависимости, interop среды и т.д.
- **Не** кладите в `runtime\`: исходники `.cs`, `.csproj`, файлы студии.
- Целевая платформа: **.NET Framework 4.8** (как ISIDA / текущая студия).
- `.pdb` — по желанию (для отладки у заказчика; на размер пакета влияет).

<!-- TODO: таблица «типичный минимум для Velum-like» — заполнить при публикации эталона velum -->

### 6.4. Подготовьте `BootData\Environment\`

Минимум для прохождения «Проверить» в студии:

**`EnvironmentRecipes.yaml`**

```yaml
# Рецепты среды (привязка к adaptive_action_id в Data агента).
recipes: []
```

**`EnvironmentTriggers.yaml`**

```yaml
# Триггеры среды: событие → influence_action_id.
triggers: []
```

Для осмысленного старта добавьте 1–2 примера с валидными `adaptive_action_id` / `influence_action_id`, которые есть в **образцах** `.dat` вашего BootData или в документации к симбионту. Иначе студия откроет YAML, но прогон сценариев может не найти ID в данных проекта.

Формат файлов — см. [`AdapterContract.md`](AdapterContract.md) § YAML среды.

### 6.5. Заполните `schema\`

Студия **не придумывает** поля — только то, что описано в JSON. Имена файлов (целевые для `contractVersion` 1.0):

| Файл | Содержание |
|------|------------|
| `recipe-preconditions.json` | Поля блока `preconditions:` рецепта |
| `recipe-steps.json` | Допустимые `type` шагов и параметры |
| `trigger-filter.json` | Фильтр документа/контекста триггера (например `document_kinds`) |
| `trigger-detect.json` | Правила `detect:` (kind, environment, command_ids, …) |

Минимальный фрагмент поля preconditions (пример):

```json
{
  "fields": [
    {
      "key": "document_kinds",
      "label": "Типы документа",
      "type": "stringList",
      "enumValues": ["part", "assembly", "drawing"],
      "required": false,
      "default": []
    }
  ]
}
```

<!-- TODO: полная схема JSON descriptor + примеры для velum — в Contract или приложение к гайду -->

**Правило:** каждый `key` в schema должен поддерживаться **вашим runtime** при чтении YAML. Иначе студия сохранит файл, а среда его проигнорирует.

### 6.6. Создайте `manifest.json`

Пример:

```json
{
  "id": "sld_adapter",
  "displayName": "SolidWorks Adapter (пример)",
  "version": "1.0.0",
  "contractVersion": "1.0",
  "author": "Your Company",
  "bootDataRelativePath": "BootData",
  "installerTemplateRelativePath": "installer/symbiont.iss",
  "schemaVersion": "1.0"
}
```

| Поле | Описание |
|------|----------|
| `id` | Каталог в `%ProgramData%\ISIDA\Adapters\{id}\`; только `[a-z0-9_-]` |
| `contractVersion` | Версия контракта платформы; студия 1.x → `1.0` |
| `bootDataRelativePath` | Обычно `BootData` |
| `installerTemplateRelativePath` | Путь к `.iss` от корня пакета |

### 6.7. `adapter-settings\Settings.xml` (если нужен host)

Шаблон настроек **среды исполнения** (куда host кладёт логи, где ищет BootData **на целевой машине после установки**). Это **не** файл проекта симбионта в студии.

<!-- TODO: перечень обязательных ключей для Velum-like адаптеров — EnvironmentRecipesFilePath, BootDataFolderPath, … -->

Пути часто задают через `%ProgramData%\...` — см. Contract.

### 6.8. `installer\symbiont.iss`

Скрипт Inno Setup: копирование `runtime\`, Data/BootData проекта симбионта, при необходимости regasm / реестр среды.

Студия вызывает `ISCC.exe` с define-параметрами (целевое):

- `/DProjectRoot=...` — корень **проекта симбионта**;
- `/DDeployDataRoot=`, `/DDeployBootData=`, … — из `Settings.xml` **проекта**;
- `/DAdapterId=...`, `/DRuntimeSource=...` — из зарегистрированного пакета в Adapters.

## 7. Регистрация и дальнейшая работа в студии

Одинаково для **своего** пакета и для **полученного** от коллеги:

1. AIStudio → **Адаптеры** → **Установить адаптер…** → папка или ZIP пакета.
2. Копия в `%ProgramData%\ISIDA\Adapters\{id}\`.
3. **Проверить** — отчёт OK / предупреждения / ошибки (см. § 9).
4. **Файл → Новый проект** (или аналог) → выбрать адаптер `{id}` → BootData из пакета в проект.
5. Редактирование гомеостаза и среды, виртуальные тесты.
6. **Собрать установщик…** — dist + ISS (§ 6.8, профиль проекта в `Settings.xml`).

Повторная установка того же `id`: замена каталога в Adapters (с подтверждением).

**Публикация пакета:** экспорт ZIP из выхода мастера «Собрать пакет…» или вручную заархивировать папку после § 6 — без отдельного «релизного» инструмента.

## 8. Два вида настроек (частая ошибка)

| Файл | Где | Кто редактирует | Зачем |
|------|-----|-----------------|-------|
| `adapter-settings\Settings.xml` | В пакете адаптера | Автор адаптера | Дефолты host-среды после установки у конечного пользователя |
| `Settings\Settings.xml` | В **проекте симбионта** | Студия / разработчик симбионта | Пути Data, BootData, `AdapterId`, шаблон ISS, куда ставить dist |

Студия **не** подменяет ваш `adapter-settings` при редактировании симбионта. Связь: установщик из студии раскладывает файлы по путям из **профиля проекта симбионта**.

---

## 9. Что делает «Проверить» (ожидаемое поведение студии)

| Проверка | Ошибка, если |
|----------|----------------|
| `manifest.json` читается, `contractVersion` поддерживается | Нет файла, неверный JSON, неизвестная версия |
| `id` валиден | Пустой id, недопустимые символы |
| Все файлы `schema\` из manifest/Contract парсятся | Отсутствует файл, битый JSON |
| `BootData\Environment\*.yaml` читаются кодеком студии | Синтаксис YAML, нет секций `recipes` / `triggers` |
| Ключи в sample YAML ⊆ ключи schema (runtime contract) | Лишние поля без schema |
| `runtime\` не пустой | Нет DLL |
| `installer\*.iss` существует, если указан в manifest | Путь битый |
| Опционально: metadata DLL | Только фаза 2 платформы; для MVP не требуется |

Типичные сообщения и исправления — § 10.

## 10. Частые ошибки

| Симптом | Вероятная причина | Что сделать |
|---------|-------------------|-------------|
| «Неподдерживаемый contractVersion» | В manifest указана версия новее студии | Поставить `1.0` или обновить студию |
| «Нет секции recipes/triggers» | Неверные имена файлов или корень YAML | Имена строго `EnvironmentRecipes.yaml`, `EnvironmentTriggers.yaml` |
| «Ключ не описан в schema» | В YAML есть поле, которого нет в JSON schema | Добавить поле в schema **и** поддержку в runtime, либо убрать из YAML |
| «Пустой runtime» | DLL не скопированы | Заполнить `runtime\` из bin сборки |
| Студия не видит адаптер после установки | Ошибка до копирования / неверный `id` | Смотреть отчёт установки; проверить `%ProgramData%\ISIDA\Adapters\{id}\` |
| Рецепты в студии OK, в среде не работают | Runtime не читает те же ключи, что schema | Сверить с `AdapterContract.md` и тестами round-trip |
| Путают с репозиторием Velum | Ожидание, что студия читает `Velum\config` | Только готовый пакет; config репозитория — дело автора при сборке |

## 11. Версионирование

- **`contractVersion` в manifest** — согласование с AIStudio (1.0 для первого релиза платформы).
- **`version` в manifest** — ваша версия пакета (`1.0.0`, `1.0.1`, …).
- Новые поля в YAML или schema → либо новая **`contractVersion`**, либо согласованное минорное расширение (см. Contract); студия не добавляет поля сама.

## 12. Чеклист перед регистрацией / публикацией пакета

- [ ] `manifest.json` с уникальным `id`
- [ ] `contractVersion`: `1.0`
- [ ] `schema\` — все четыре файла (или явно согласованный минимум в Contract)
- [ ] `runtime\` — полный набор DLL для чистой машины (проверено вручную)
- [ ] `BootData\Environment\EnvironmentRecipes.yaml` и `EnvironmentTriggers.yaml`
- [ ] `installer\symbiont.iss` (если сборка из студии)
- [ ] `adapter-settings\Settings.xml` (если host его использует)
- [ ] Пример YAML проходит ваш runtime на тестовой машине
- [ ] ZIP/папка без мусора (`.git`, `obj`, лишние репозитории)
- [ ] При передаче коллеге — та же `contractVersion`, что у его версии студии

## Приложение A. Пример: пакет после сборки (Velum-like)

Иллюстрация, **не** требование копировать из репозитория Velum:

1. Сборка → `Velum\bin\Debug\`.

5. В студии: **Собрать пакет…** (когда есть) или вручную → **Установить** → `Adapters\velum\`.
